UNIT I - Introduction and Process Management

1. Types of Operating Systems

1.1 Simple Batch Systems

Key Points:
  • Early computer systems operated by operators only
  • No user interaction during execution
  • Jobs submitted using punch cards
  • High turnaround time

Characteristics:

  • Job Submission: Programs, input data, and control information submitted as punch cards
  • Sequential Execution: Jobs executed one after another
  • No Interaction: Users cannot interact with programs during execution
  • Automatic Job Sequencing: OS automatically transfers control from one job to the next

Simple Batch System Structure

User → [Punch Cards] → Operator → [Job Queue] → CPU → [Output] → User
                                      ↓
                                 [OS in Memory]

Problems:

  • CPU Idle Time: CPU much faster than I/O devices
  • Poor Resource Utilization: CPU waits for I/O operations
  • Solution - Spooling: Simultaneous Peripheral Operation On-Line

1.2 Multiprogrammed Batch Systems

Evolution from Simple Batch: Multiple jobs kept in memory simultaneously to increase CPU utilization

Key Concepts:

  • Job Pool: Collection of jobs stored on disk ready for execution
  • Memory Management: Multiple jobs reside in main memory
  • CPU Switching: When one job waits for I/O, CPU switches to another job
  • Job Scheduling: OS decides which job to load into memory
  • CPU Scheduling: OS decides which ready job gets CPU

Multiprogramming Concept

Memory Layout:
┌─────────────┐
│     OS      │
├─────────────┤
│   Job 1     │ ← Currently executing
├─────────────┤
│   Job 2     │ ← Waiting for I/O
├─────────────┤
│   Job 3     │ ← Ready to execute
└─────────────┘

CPU switches between jobs to maximize utilization

1.3 Time-Sharing Systems

Interactive Computing: Extension of multiprogramming where CPU switches between jobs very frequently

Characteristics:

  • Interactive: Direct communication between user and computer
  • Short Response Time: Few seconds or less
  • Online File System: Files stored and accessed online
  • Multiple Users: Many users can use system simultaneously
Time-Sharing vs Batch Systems:
Aspect Batch Systems Time-Sharing Systems
User Interaction No interaction Interactive
Response Time Long turnaround time Short response time
Suitable for Large jobs Short commands
Debugging Static debugging Interactive debugging

1.4 Personal Computer Systems

  • Single User: Dedicated to individual users
  • User Convenience: Focus on ease of use rather than CPU utilization
  • Evolution: From simple single-tasking to modern multitasking systems
  • Examples: MS-DOS → Windows, Macintosh OS

1.5 Parallel Systems

Multiprocessor Systems: More than one processor sharing bus, clock, memory, and peripherals

Advantages:

  • Increased Throughput: More work in less time
  • Economy of Scale: Shared peripherals and resources
  • Increased Reliability: Graceful degradation

Types:

  • Symmetric Multiprocessing (SMP): All processors run identical OS copies
  • Asymmetric Multiprocessing: Master-slave relationship

SMP vs AMP

Symmetric Multiprocessing (SMP):
CPU1 ←→ [Shared Memory] ←→ CPU2
  ↕                        ↕
[I/O]                    [I/O]

Asymmetric Multiprocessing (AMP):
Master CPU → [Controls] → Slave CPUs
     ↓
[All I/O Operations]

1.6 Distributed Systems

Loosely Coupled: Processors don't share memory or clock, communicate via networks

Advantages:

  • Resource Sharing: Access remote resources
  • Computation Speedup: Load sharing across sites
  • Reliability: Fault tolerance through redundancy
  • Communication: Information exchange between sites

1.7 Real-Time Systems

Time Constraints: Correctness depends on producing results within specified time limits

Types:

  • Hard Real-Time: Guarantee critical tasks complete within deadlines
  • Soft Real-Time: Priority to critical tasks but no guarantee
Real-Time System Applications:
  • Industrial control systems
  • Medical imaging systems
  • Automobile engine control
  • Robotics
  • Defense systems

2. OS Structure

2.1 System Components

Process Management:

  • Create and delete processes

  • Process synchronization and communication
  • Deadlock handling

Main Memory Management:

  • Track memory usage
  • Allocate and deallocate memory
  • Decide which processes to load

File Management:

  • Create and delete files and directories
  • Support primitives for file manipulation
  • Map files to secondary storage
  • Backup files

I/O System Management:

  • Buffer-cache system
  • General device-driver interface
  • Drivers for specific hardware devices

Simple Structure (MS-DOS):

  • No clear modular design
  • Applications can access hardware directly
  • Difficult to maintain and extend

  • Layer N: User interface

3. Process Management

3.1 Process Concept

Process: A program in execution. It includes program code, current activity, program counter, and registers.

Process vs Program:

  • Program: Passive entity (file on disk)
  • Process: Active entity with program counter and resources

Process Components:

  • Text Section: Program code
  • Data Section: Global variables
  • Heap: Dynamically allocated memory
  • Stack: Temporary data (function parameters, return addresses, local variables)

3.2 Process State

Process States:

  • New: Process is being created
  • Ready: Process is waiting to be assigned to processor
  • Running: Instructions are being executed
  • Waiting: Process is waiting for some event (I/O completion)
  • Terminated: Process has finished execution

3.3 Process Control Block (PCB)

PCB: Data structure containing all information about a process

PCB Contents:

  • Process State: Current state of process
  • Program Counter: Address of next instruction
  • CPU Registers: Accumulators, index registers, stack pointers
  • CPU Scheduling Information: Priority, scheduling queue pointers
  • Memory Management Information: Base/limit registers, page tables
  • Accounting Information: CPU time used, time limits
  • I/O Status Information: List of I/O devices, open files

3.4 Process Scheduling

Objectives:

  • Multiprogramming: Keep CPU busy at all times
  • Time-sharing: Switch CPU frequently for user interaction

Scheduling Queues:

  • Job Queue: All processes in the system
  • Ready Queue: Processes in memory, ready to execute
  • Device Queues: Processes waiting for I/O devices

Process Scheduling Diagram

[Job Queue] → [Long-term Scheduler] → [Ready Queue]
                                           ↓
                                    [CPU Scheduler]
                                           ↓
                                        [CPU] → [Terminate]
                                           ↓
                                    [I/O Queue] → [I/O] → [Ready Queue]

Types of Schedulers:

  • Long-term Scheduler (Job Scheduler):
    • Selects processes from mass storage
    • Controls degree of multiprogramming
    • Executes infrequently
  • Short-term Scheduler (CPU Scheduler):
    • Selects process from ready queue
    • Executes very frequently (milliseconds)
    • Must be extremely fast
  • Medium-term Scheduler:
    • Swapping - removes processes from memory
    • Reduces degree of multiprogramming

3.5 Context Switch

Context Switch: Switching CPU from one process to another by saving and restoring process states

Steps:

  1. Save state of current process in its PCB
  2. Load saved state of new process from its PCB
  3. Resume execution of new process

Context Switch Time:

  • Pure overhead - no useful work done
  • Depends on hardware support
  • Typical time: 1-1000 microseconds

3.6 Cooperating Processes

Types of Processes:

  • Independent Process: Cannot affect or be affected by others
  • Cooperating Process: Can affect or be affected by other processes

Reasons for Cooperation:

  • Information sharing
  • Computation speedup
  • Modularity
  • Convenience
Producer-Consumer Problem:
  • Producer: Generates data
  • Consumer: Consumes data
  • Buffer: Shared storage (bounded/unbounded)
  • Synchronization: Required to prevent race conditions

3.7 Threads

Thread: Lightweight process - allows multiple execution paths within a single process

Benefits:

  • Responsiveness
  • Resource sharing
  • Economy
  • Utilization of multiprocessor architectures

4. CPU Scheduling

4.1 Basic Concepts

CPU-I/O Burst Cycle:

  • Process execution alternates between CPU burst and I/O burst
  • CPU-bound processes: Long CPU bursts
  • I/O-bound processes: Short CPU bursts

CPU Burst Distribution

Frequency

    │ ████
    │ ████
    │ ████ ██
    │ ████ ██ █
    │ ████ ██ █ █
    └─────────────────→ CPU Burst Duration
    Many short bursts, few long bursts

4.2 CPU Scheduler

CPU Scheduler: Selects a process from ready queue when CPU becomes idle

Scheduling Decisions occur when:

  1. Process switches from running to waiting (I/O request)
  2. Process switches from running to ready (interrupt)
  3. Process switches from waiting to ready (I/O completion)
  4. Process terminates

Types:

  • Nonpreemptive: Scheduling only in cases 1 and 4
  • Preemptive: Scheduling in all four cases

4.3 Scheduling Criteria

Criterion Description Goal
CPU Utilization Percentage of time CPU is busy Maximize
Throughput Number of processes completed per unit time Maximize
Turnaround Time Time from submission to completion Minimize
Waiting Time Time spent waiting in ready queue Minimize
Response Time Time from submission to first response Minimize

4.4 Scheduling Algorithms

4.4.1 First-Come, First-Served (FCFS)

  • Simplest scheduling algorithm
  • Nonpreemptive
  • Poor performance due to convoy effect
FCFS Example:
Process Burst Time
P124
P23
P33

Gantt Chart: P1 (0-24) | P2 (24-27) | P3 (27-30)

Average Waiting Time = (0 + 24 + 27) / 3 = 17 ms

4.4.2 Shortest-Job-First (SJF)

  • Optimal algorithm - minimizes average waiting time
  • Can be preemptive or nonpreemptive
  • Difficulty: predicting next CPU burst

CPU Burst Prediction:

τ(n+1) = α × t(n) + (1-α) × τ(n) Where: - τ(n+1) = predicted next burst - t(n) = actual last burst - α = weighting factor (0 ≤ α ≤ 1)

4.4.3 Priority Scheduling

  • Each process assigned a priority
  • CPU allocated to highest priority process
  • Problem: Starvation of low-priority processes
  • Solution: Aging - gradually increase priority

4.4.4 Round Robin (RR)

  • Designed for time-sharing systems
  • Each process gets CPU for time quantum
  • Preemptive FCFS with time slicing
Time Quantum Selection:
  • Too large → behaves like FCFS
  • Too small → too many context switches
  • Rule: 80% of CPU bursts should be shorter than quantum

4.4.5 Multilevel Queue Scheduling

  • Ready queue divided into separate queues
  • Processes permanently assigned to queues
  • Each queue has its own scheduling algorithm
  • Scheduling between queues (usually priority-based)

Multilevel Queue Structure

┌─────────────────────┐
│  System Processes   │ ← Highest Priority
├─────────────────────┤
│ Interactive Processes│
├─────────────────────┤
│ Interactive Editing │
├─────────────────────┤
│  Batch Processes    │
├─────────────────────┤
│ Student Processes   │ ← Lowest Priority
└─────────────────────┘

4.4.6 Multilevel Feedback Queue

  • Processes can move between queues
  • Aging prevents starvation
  • Separates processes based on CPU burst characteristics
  • Most general and flexible

UNIT II - Process Synchronization and Deadlocks

1. Process Synchronization

1.1 Background

Race Condition: Multiple processes access shared data concurrently, and the outcome depends on the order of execution

Producer-Consumer Problem:

Producer: while (true) { // produce item; while (count == BUFFER_SIZE); // busy wait; buffer[in] = item; in = (in + 1) % BUFFER_SIZE; count++; } Consumer: while (true) { while (count == 0); // busy wait; item = buffer[out]; out = (out + 1) % BUFFER_SIZE; count--; // consume item; }

Problem with Concurrent Execution:

  • count++ and count-- are not atomic operations
  • Machine code involves: load, increment/decrement, store
  • Interleaving can cause incorrect results

1.2 Critical-Section Problem

Critical Section: Segment of code where shared variables are accessed

General Structure of Process

do {
    [Entry Section];
    
    Critical Section;
    
    [Exit Section];
    
    Remainder Section;
    
} while (true);

Requirements for Solution:

  1. Mutual Exclusion: Only one process in critical section at a time
  2. Progress: Selection of next process cannot be postponed indefinitely
  3. Bounded Waiting: Limit on how many times others can enter before a waiting process

1.3 Two-Process Solutions

Algorithm 1 - Turn Variable:

Shared variable: int turn = 0; Process P0: do { while (turn != 0); critical section; turn = 1; remainder section; } while (true); Process P1: do { while (turn != 1); critical section; turn = 0; remainder section; } while (true);
  • ✓ Mutual exclusion
  • ✗ Progress (strict alternation)

Algorithm 2 - Flag Array:

Shared variable: boolean flag[2] = {false, false}; Process Pi: do { flag[i] = true; while (flag[j]); critical section; flag[i] = false; remainder section; } while (true);
  • ✗ Mutual exclusion (both can enter)
  • ✓ Progress

Algorithm 3 - Peterson's Solution:

Shared variables: boolean flag[2] = {false, false}; int turn = 0; Process Pi: do { flag[i] = true; turn = j; while (flag[j] && turn == j); critical section; flag[i] = false; remainder section; } while (true);
  • ✓ Mutual exclusion
  • ✓ Progress
  • ✓ Bounded waiting

1.4 Multiple-Process Solutions

Bakery Algorithm:

  • Based on bakery number system
  • Each process gets a number
  • Process with smallest number enters first
  • Handles n processes
Shared data: boolean choosing[n] = {false}; int number[n] = {0}; Process Pi: do { choosing[i] = true; number[i] = max(number[0],...,number[n-1]) + 1; choosing[i] = false; for (j = 0; j < n; j++) { while (choosing[j]); while ((number[j] != 0) && ((number[j], j) < (number[i], i))); } critical section; number[i] = 0; remainder section; } while (true);

1.5 Synchronization Hardware

TestAndSet Instruction:

boolean TestAndSet(boolean *target) { boolean rv = *target; *target = true; return rv; } // Usage for mutual exclusion Shared variable: boolean lock = false; Process Pi: do { while (TestAndSet(&lock)); critical section; lock = false; remainder section; } while (true);

Swap Instruction:

void Swap(boolean *a, boolean *b) { boolean temp = *a; *a = *b; *b = temp; } // Usage for mutual exclusion Shared variable: boolean lock = false; Process Pi: do { key = true; while (key == true) Swap(&lock, &key); critical section; lock = false; remainder section; } while (true);
Hardware Solutions:
  • ✓ Simple and effective
  • ✓ Work on multiprocessors
  • ✗ Busy waiting (spinlock)
  • ✗ Starvation possible

1.6 Semaphores

Semaphore: Integer variable accessed only through atomic wait() and signal() operations
wait(S) { while (S <= 0); // busy wait; S--; } signal(S) { S++; }

Types of Semaphores:

  • Binary Semaphore: Values 0 or 1 (mutex)
  • Counting Semaphore: Unrestricted integer values

Usage:

  • Mutual Exclusion: Initialize semaphore to 1
  • Synchronization: Initialize semaphore to 0
Mutual Exclusion with Semaphores:
Semaphore mutex = 1; Process Pi: do { wait(mutex); critical section; signal(mutex); remainder section; } while (true);

Implementation without Busy Waiting:

typedef struct { int value; struct process *list; } semaphore; wait(semaphore *S) { S->value--; if (S->value < 0) { add this process to S->list; block(); } } signal(semaphore *S) { S->value++; if (S->value <= 0) { remove a process P from S->list; wakeup(P); } }

1.7 Deadlocks and Starvation

Deadlock Example:

Process P0: Process P1: do { do { wait(S); wait(Q); wait(Q); wait(S); ...; ...; signal(S); signal(Q); signal(Q); signal(S); } while (true); } while (true);

Starvation (Indefinite Blocking):

  • Process waits indefinitely in semaphore queue
  • Can occur with LIFO queue implementation
  • Solution: FIFO queue or aging

1.8 Critical Regions

Critical Region: High-level synchronization construct to avoid semaphore programming errors
region v when B do S; Where: - v: shared variable; - B: boolean expression; - S: statement;

Advantages:

  • Automatic mutual exclusion
  • Prevents common programming errors
  • Compiler enforced

1.9 Monitors

Monitor: High-level synchronization construct that combines data, operations, and synchronization in one module

Monitor Structure

monitor MonitorName {
    // shared variable declarations;
    
    procedure P1(...) {
        ...;
    }
    
    procedure P2(...) {
        ...;
    }
    
    initialization code;
}

Characteristics:

  • Only one process active in monitor at a time
  • Shared variables accessible only within monitor
  • Automatic mutual exclusion

Condition Variables:

condition x; x.wait(); // suspend process; x.signal(); // resume one suspended process;
Dining Philosophers with Monitors:
monitor DiningPhilosophers { enum {THINKING, HUNGRY, EATING} state[5]; condition self[5]; void pickup(int i) { state[i] = HUNGRY; test(i); if (state[i] != EATING) self[i].wait(); } void putdown(int i) { state[i] = THINKING; test((i + 4) % 5); test((i + 1) % 5); } void test(int i) { if ((state[(i + 4) % 5] != EATING) && (state[i] == HUNGRY) && (state[(i + 1) % 5] != EATING)) { state[i] = EATING; self[i].signal(); } } }

2. Deadlocks

2.1 System Model

Deadlock: Set of processes blocked forever, each waiting for a resource held by another process in the set

Resource Usage:

  1. Request: Process requests resource
  2. Use: Process uses resource
  3. Release: Process releases resource

2.2 Deadlock Characterization

Necessary Conditions (All must hold):

  1. Mutual Exclusion: At least one resource must be non-sharable
  2. Hold and Wait: Process holds resources while waiting for others
  3. No Preemption: Resources cannot be forcibly taken away
  4. Circular Wait: Circular chain of processes waiting for resources

Resource-Allocation Graph

Process → Resource (Request Edge)
Resource → Process (Assignment Edge)

Example:
P1 → R1 → P2 → R2 → P3 → R3 → P1 (Deadlock!)

Graph Analysis:

  • No cycle → No deadlock
  • Cycle + single instance per resource → Deadlock
  • Cycle + multiple instances → Deadlock possible

2.3 Methods for Handling Deadlocks

  1. Deadlock Prevention: Ensure at least one necessary condition never holds
  2. Deadlock Avoidance: Dynamically examine resource allocation state
  3. Deadlock Detection and Recovery: Allow deadlocks, detect and recover
  4. Ignore the Problem: Assume deadlocks are rare (used by UNIX)

2.4 Deadlock Prevention

Attacking Mutual Exclusion:

  • Not possible for non-sharable resources
  • Some resources are inherently non-sharable

Attacking Hold and Wait:

  • Request all resources at once before execution
  • Release all resources before requesting new ones
  • Disadvantages: Low resource utilization, starvation

Attacking No Preemption:

  • If request cannot be granted, release all held resources
  • Works only for resources with savable state (CPU, memory)
  • Not applicable to printers, tape drives

Attacking Circular Wait:

  • Impose total ordering on resource types
  • Request resources in increasing order
  • Prevents circular wait condition
Resource Ordering Example:
  • F(tape drive) = 1
  • F(disk drive) = 5
  • F(printer) = 12

Process must request tape drive before disk drive before printer

2.5 Deadlock Avoidance

Safe State: System can allocate resources to each process in some order and avoid deadlock

Safe vs Unsafe States:

  • Safe state → No deadlock
  • Unsafe state → Deadlock possible
  • All deadlock states are unsafe
  • Not all unsafe states are deadlock

State Space Diagram

┌─────────────────────────────────┐
│          All States             │
│  ┌─────────────────────────┐    │
│  │      Safe States        │    │
│  │                         │    │
│  └─────────────────────────┘    │
│  ┌─────────────────────────┐    │
│  │     Unsafe States       │    │
│  │  ┌─────────────────┐    │    │
│  │  │ Deadlock States │    │    │
│  │  └─────────────────┘    │    │
│  └─────────────────────────┘    │
└─────────────────────────────────┘

Banker's Algorithm:

Analogy: Bank never allocates cash such that it cannot satisfy maximum needs of all customers

Data Structures:

  • Available[m]: Available instances of each resource
  • Max[n][m]: Maximum demand of each process
  • Allocation[n][m]: Currently allocated resources
  • Need[n][m]: Remaining resource needs (Max - Allocation)

Safety Algorithm:

1. Work = Available; Finish[i] = false for all i; 2. Find i such that: Finish[i] == false AND Need[i] <= Work; 3. Work = Work + Allocation[i]; Finish[i] = true; 4. If all Finish[i] == true, system is safe;

Resource-Request Algorithm:

Request[i] = request vector for process Pi; 1. If Request[i] <= Need[i], go to step 2; Otherwise, error (exceeds maximum claim); 2. If Request[i] <= Available, go to step 3; Otherwise, wait (resources not available); 3. Pretend allocation: Available = Available - Request[i]; Allocation[i] = Allocation[i] + Request[i]; Need[i] = Need[i] - Request[i]; 4. Run safety algorithm: If safe → grant request; If unsafe → rollback and wait;

2.6 Deadlock Detection

Single Instance - Wait-for Graph:

  • Remove resource nodes from resource-allocation graph
  • Pi → Pj if Pi waits for resource held by Pj
  • Cycle in wait-for graph → deadlock

Multiple Instances - Detection Algorithm:

1. Work = Available; Finish[i] = false if Allocation[i] != 0; 2. Find i such that: Finish[i] == false AND Request[i] <= Work; 3. Work = Work + Allocation[i]; Finish[i] = true; 4. If Finish[i] == false for some i, then system is deadlocked;

When to Invoke Detection:

  • Every time request cannot be granted immediately
  • Periodically (e.g., once per hour)
  • When CPU utilization drops below threshold

2.7 Recovery from Deadlock

Process Termination:

  • Abort all deadlocked processes: Expensive but effective
  • Abort one process at a time: Overhead of repeated detection

Victim Selection Factors:

  • Priority of process
  • Computation time and remaining time
  • Resources used and needed
  • Number of processes to terminate
  • Interactive vs batch process

Resource Preemption:

  1. Selecting a victim: Minimize cost
  2. Rollback: Return to safe state
  3. Starvation: Ensure same process not always chosen
Deadlock Handling Summary:
Method Approach Overhead When to Use
Prevention Deny necessary conditions High Simple systems
Avoidance Safe state maintenance Medium Known resource needs
Detection Detect and recover Low Rare deadlocks
Ignore Do nothing None Very rare deadlocks
